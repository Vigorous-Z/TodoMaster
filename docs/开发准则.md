## TodoMaster Vibe 开发准则 v1.0

### 1. 核心原则

**“能跑的不动，要动的想好，想不好的先问。”**


### 2. 禁止破坏性更新（金科玉律）

以下行为 **必须经过确认后方可执行**，任何 AI 不得擅自进行：

| 禁止行为 | 示例 | 正确处理方式 |
| :--- | :--- | :--- |
| **删除现有代码** | 删除某个函数、组件、文件 | 先询问“这个函数还有用吗？是否可以删除？” |
| **重命名现有标识符** | 改函数名、变量名、组件名、CSS 类名 | 除非用户明确要求，否则不重命名 |
| **修改现有接口签名** | 改 `api_get_tasks()` 的入参或出参格式 | 新增接口或新增字段，不修改已有字段 |
| **修改现有数据模型** | 改 `tasks` 表的字段类型或含义 | 新增字段，不修改已有字段的含义 |
| **更改现有 UI 交互逻辑** | 改“回车创建任务”为“Ctrl+回车” | 新增快捷键，不改变既有交互 |
| **重构已有功能** | “我觉得这里可以优化一下，我重构了” | **坚决禁止！** 除非明确要求重构 |


### 3. 剃刀原则（奥卡姆剃刀）

**“如无必要，勿增实体。”**

做任何决策时，遵循以下优先级：

1. **复用优先**：先找有没有现成的函数、组件、方案可以用。
2. **扩展优于新增**：能加参数解决的问题，不新建系统。
3. **简单优于复杂**：能 10 行解决的问题，不用 50 行。
4. **明确优于花哨**：代码清晰、可读性强，比看起来厉害更重要。

**反例**：
- 明明可以加个字段解决的问题，非要建一张新表。
- 明明可以在现有组件上加个 prop，非要新建一个组件。
- 明明可以用简单 `if-else`，非要引入设计模式。


### 4. 不确定先询问

当遇到以下情况时，**先暂停，发起澄清，不猜、不编、不擅自假设**：

| 情况 | 示例 | 应如何做 |
| :--- | :--- | :--- |
| **需求理解不清** | “用户想要 AI 解析，但不知道输出格式应该是什么” | 提问：“AI 解析输出的 JSON 格式是什么？” |
| **实现方案不明确** | “这个功能是放前端还是放后端？” | 提问：“这个功能应该放在前端还是后端？” |
| **技术选型不确定** | “用什么库来实现这个功能？” | 提问：“我应该用什么库来实现？” |
| **UI/UX 不明确** | “点击后应该直接跳转还是弹窗？” | 提问：“点击后应该跳转还是弹窗？” |
| **边界情况不明确** | “如果用户没登录，这个功能应该灰显还是隐藏？” | 提问：“未登录时这个功能应该灰显还是隐藏？” |

**提问标准格式**：

```
【需要澄清】
问题：xxx
可选项：A / B / C
我的建议：选 A，因为 xxx
请确认。
```


### 5. 风格统一

#### 5.1 代码风格

| 规范 | 要求 |
| :--- | :--- |
| **Vue 组件** | 使用 `<script setup>` 语法，组合式 API |
| **命名** | 组件 PascalCase，变量/函数 camelCase，CSS 类名 kebab-case |
| **导入顺序** | Vue / Pinia → 第三方库 → 本地组件 → 本地 Store → 工具函数 |
| **注释** | 关键函数和复杂逻辑必须加中文注释，简单逻辑不加 |
| **代码长度** | 函数不超过 80 行，超过则拆分；单行不超过 100 字符 |
| **缩进** | 前端 2 空格，后端 4 空格 |

#### 5.2 UI 风格

| 规范 | 要求 |
| :--- | :--- |
| **颜色** | 使用 CSS 变量（`var(--accent)`），不硬编码色值 |
| **间距** | 使用 4 的倍数（4px、8px、12px、16px、20px、24px） |
| **字体** | 系统字体栈，不使用自定义字体 |
| **圆角** | 使用 `var(--radius-sm)` / `var(--radius-md)` |
| **动画** | 使用 `var(--transition)`（120ms ease），不额外加动画 |
| **新增 UI** | 参照现有组件风格，保持视觉一致性 |

#### 5.3 提交规范

```
feat: 新增功能（如“新增 AI 解析输入框”）
fix: 修复 Bug（如“修复任务列表排序错误”）
style: 样式调整（如“调整侧边栏间距”）
refactor: 代码重构（需提前确认）
docs: 文档更新
chore: 构建/工具配置
```

> **禁止**：一次性提交多个不相关的改动。每个提交只做一件事。


### 6. 开发流程

```
1. 收到需求 → 先判断：是否影响现有功能？
   ↓
2. 有影响 → 先提问澄清，确认后再动手
   无影响 → 继续
   ↓
3. 写代码（遵循剃刀原则，复用优先）
   ↓
4. 自检：是否改动了不该改的地方？
   ↓
5. 提交（规范提交信息）
```


### 7. 反面案例警示

| 警示 | 说明 |
| :--- | :--- |
| ❌ 不要“顺便重构” | “我看见有段代码可以优化，顺手改了” → **严禁！** |
| ❌ 不要“顺手改格式” | “我看这里缩进不对，改一下” → 除非专门做格式化任务 |
| ❌ 不要“补充文档” | “我顺便把这个函数也加个注释” → 无关任务不做 |
| ❌ 不要“增强一下” | “这个功能我觉得还可以增加 xxx 特性” → 先问再动手 |
| ❌ 不要“整体优化” | “我把这个模块重新设计一下，更好” → **绝对禁止！** |

> **核心精神**：来的时候什么样，走的时候也什么样。只改任务要求的部分。


### 8. 总结（记忆口诀）

```
③ 想不好的先问
④ 问清楚了再做
⑤ 做了只改该改的
```
